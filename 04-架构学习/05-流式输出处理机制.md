# 流式输出处理机制

## 一、流式数据的接收位置

### 双层循环中的流式数据流

```
外层循环 (packages/opencode/src/session/prompt.ts:257)
  │
  ├─ 决策：何时调用 LLM
  │  └─ 创建 SessionProcessor
  │
  └─ 调用 processor.process() → 进入内层循环

内层循环 (packages/opencode/src/session/processor.ts:44)
  │
  ├─ 调用 LLM.stream() 获取流
  │  └─ const stream = await LLM.stream(streamInput)
  │
  └─ for await (const value of stream.fullStream)
     └─ 处理每个流式事件（数据可能不完整）
```

**关键点**：流式数据只在**内层循环**中接收，外层循环只负责决策和任务管理。

## 二、不完整数据的处理策略

### 1. 文本输出的增量累积

**事件序列**：

```
text-delta (1) → text-delta (2) → ... → text-delta (N) → text-end
```

**处理代码**：

```typescript
case "text-start":
  currentText = {
    id: Identifier.ascending("part"),
    messageID: input.assistantMessage.id,
    sessionID: input.assistantMessage.sessionID,
    type: "text",
    text: "",
    time: {
      start: Date.now(),
    },
    metadata: value.providerMetadata,
  }
  break

case "text-delta":
  if (currentText) {
    currentText.text += value.text
    if (value.providerMetadata) currentText.metadata = value.providerMetadata
    if (currentText.text)
      await Session.updatePart({
        part: currentText,
        delta: value.text,
      })
  }
  break

case "text-end":
  // 文本完整，保存最终状态
  if (currentText) {
    currentText.text = currentText.text.trimEnd()
    currentText.time = {
      start: Date.now(),
      end: Date.now(),
    }
    if (value.providerMetadata) currentText.metadata = value.providerMetadata
    await Session.updatePart(currentText)
  }
  currentText = undefined
  break
```

**状态转换**：

```
空文本 "" → 增量1 "Hel" → 增量2 "lo " → 增量3 "World" → 完整 "Hello World"
```

### 2. 推理过程的增量累积

**事件序列**：

```
reasoning-start → reasoning-delta (1) → ... → reasoning-delta (N) → reasoning-end
```

**处理代码**：

```typescript
case "reasoning-start":
  if (value.id in reasoningMap) {
    continue
  }
  reasoningMap[value.id] = {
    id: Identifier.ascending("part"),
    messageID: input.assistantMessage.id,
    sessionID: input.assistantMessage.sessionID,
    type: "reasoning",
    text: "",
    time: {
      start: Date.now(),
    },
    metadata: value.providerMetadata,
  }
  break

case "reasoning-delta":
  if (value.id in reasoningMap) {
    const part = reasoningMap[value.id]
    part.text += value.text
    if (value.providerMetadata) part.metadata = value.providerMetadata
    if (part.text) await Session.updatePart({ part, delta: value.text })
  }
  break

case "reasoning-end":
  // 推理完整，保存最终状态
  if (value.id in reasoningMap) {
    const part = reasoningMap[value.id]
    part.text = part.text.trimEnd()
    part.time = {
      ...part.time,
      end: Date.now(),
    }
    if (value.providerMetadata) part.metadata = value.providerMetadata
    await Session.updatePart(part)
    delete reasoningMap[value.id]
  }
  break
```

### 3. 工具调用的多阶段状态管理

**完整事件序列**：

```
tool-input-start → tool-call → tool-result
```

**状态机转换**：

```typescript
case "tool-input-start":
  // 阶段 1: 工具调用开始，参数为空
  const part = await Session.updatePart({
    id: toolcalls[value.id]?.id ?? Identifier.ascending("part"),
    messageID: input.assistantMessage.id,
    type: "tool",
    tool: value.toolName,
    callID: value.id,
    state: {
      status: "pending",      // ← 待定状态
      input: {},              // ← 参数为空
      raw: "",                // ← 原始文本为空
    },
  })
  toolcalls[value.id] = part as MessageV2.ToolPart
  break

case "tool-input-delta":
  // 阶段 2: 参数逐步累积（可选事件，当前版本不处理）
  // 注意：在当前实现中，此事件是空处理，直接跳过
  break

case "tool-call":
  // 阶段 3: 参数完整，执行工具
  const match = toolcalls[value.toolCallId]
  if (match) {
    await Session.updatePart({
      ...match,
      tool: value.toolName,
      state: {
        status: "running",      // ← 运行中状态
        input: value.input,     // ← 完整参数已解析
        time: {
          start: Date.now(),
        },
      },
      metadata: value.providerMetadata,
    })

    // Doom Loop 检测
    const parts = await MessageV2.parts(input.assistantMessage.id)
    const lastThree = parts.slice(-3)

    if (
      lastThree.length === 3 &&
      lastThree.every(
        (p) =>
          p.type === "tool" &&
          p.tool === value.toolName &&
          p.state.status !== "pending" &&
          JSON.stringify(p.state.input) === JSON.stringify(value.input),
      )
    ) {
      // 触发警告
      await PermissionNext.ask({
        permission: "doom_loop",
        patterns: [value.toolName],
        sessionID: input.assistantMessage.sessionID,
        metadata: { tool: value.toolName, input: value.input },
        always: [value.toolName],
        ruleset: agent.permission,
      })
    }
  }
  break

          case "tool-result":
            // 工具执行结果（状态完成）
            const match = toolcalls[value.toolCallId]
            if (match && match.state.status === "running") {
              await Session.updatePart({
                ...match,
                state: {
                  status: "completed",
                  input: value.input,
                  output: value.output.output,
                  metadata: value.output.metadata,
                  title: value.output.title,
                  time: {
                    start: match.state.time.start,
                    end: Date.now(),
                  },
                  attachments: value.output.attachments,
                },
              })
              delete toolcalls[value.toolCallId]
            }
            break
```

**状态转换图**：

```
pending (tool-input-start)
    ↓ [参数解析完成]
    ↓
running (tool-call)
    ↓ [工具执行中]
    ↓
completed (tool-result)
    ↓ [保存结果，从 toolcalls 移除]
    ↓
从 UI/内存中删除
```

## 三、不完整数据的处理机制对比

| 数据类型     | 事件类型           | 不完整特征 | 处理方式                      | 存储位置                 |
| ------------ | ------------------ | ---------- | ----------------------------- | ------------------------ |
| **文本**     | `text-delta`       | 部分字符   | `text += delta` + 实时保存    | MessagePart.text         |
| **推理**     | `reasoning-delta`  | 部分内容   | `text += delta` + 元数据      | MessagePart.text         |
| **工具参数** | `tool-input-start` | 参数为空   | 创建 pending 状态，`raw = ""` | MessagePart.state.raw    |
| **工具调用** | `tool-call`        | 参数完整   | 解析 input + 执行             | MessagePart.state.input  |
| **工具结果** | `tool-result`      | 执行完成   | 保存 output + 时间戳          | MessagePart.state.output |

## 四、实时保存的 UI 显示机制

### 1. 数据流向

```
LLM 流式输出
    ↓
for await (const value of stream.fullStream)
    ↓
Session.updatePart({ part, delta: value.text })
    ↓
数据库更新 + 发布事件 (Bus.publish)
    ↓
UI 监听事件 → 实时渲染
```

### 2. 用户看到的效果

**场景：LLM 调用 `read` 工具读取文件**

| 时间 | UI 显示                         | 数据状态                             |
| ---- | ------------------------------- | ------------------------------------ |
| T0   | 无                              | 初始状态                             |
| T1   | `🔍 read(file.ts) [pending...]` | `tool-input-start`，status = pending |
| T2   | `🔍 read(file.ts) [running...]` | `tool-call`，status = running        |
| T3   | `✅ read(file.ts) [completed]`  | `tool-result`，status = completed    |

**场景：LLM 生成文本回复**

| 时间 | UI 显示 | 数据状态                       |
| ---- | ------- | ------------------------------ |
| T0   | 无      | 初始状态                       |
| T1   | `H`     | `text-delta("H")`              |
| T2   | `He`    | `text-delta("e")`              |
| T3   | `Hel`   | `text-delta("l")`              |
| T4   | `Hell`  | `text-delta("l")`              |
| T5   | `Hello` | `text-delta("o")` + `text-end` |

### 3. 为什么需要实时保存？

1. **用户体验**：用户看到"正在思考"或"正在调用工具"的实时进度，而不是白屏
2. **错误恢复**：如果进程崩溃，已经处理的部分数据不会丢失
3. **多端同步**：多个客户端可以实时看到相同的进度
4. **调试便利**：可以查看每个事件的执行时间戳和状态

## 五、完整的事件处理流程

```typescript
async process(streamInput: LLM.StreamInput) {
  const toolcalls: Record<string, MessageV2.ToolPart> = {}
  let blocked = false
  let attempt = 0

  while (true) {
    try {
      let currentText: MessageV2.TextPart | undefined
      let reasoningMap: Record<string, MessageV2.ReasoningPart> = {}

      // 调用 LLM 流式接口
      const stream = await LLM.stream(streamInput)

      // 处理 LLM 的流式输出
      for await (const value of stream.fullStream) {
        input.abort.throwIfAborted()

        switch (value.type) {
          case "start":
            SessionStatus.set(input.sessionID, { type: "busy" })
            break

          case "reasoning-delta":
            // 推理内容增量
            if (value.id in reasoningMap) {
              const part = reasoningMap[value.id]
              part.text += value.text
              if (value.providerMetadata) part.metadata = value.providerMetadata
              if (part.text) await Session.updatePart({ part, delta: value.text })
            }
            break

        case "text-delta":
          // 文本增量
          if (currentText) {
            currentText.text += value.text
            if (value.providerMetadata) currentText.metadata = value.providerMetadata
            if (currentText.text)
              await Session.updatePart({ part: currentText, delta: value.text })
          }
          break

          case "tool-input-start":
            // 工具调用开始（参数不完整）
            const part = await Session.updatePart({
              id: toolcalls[value.id]?.id ?? Identifier.ascending("part"),
              messageID: input.assistantMessage.id,
              type: "tool",
              tool: value.toolName,
              callID: value.id,
              state: {
                status: "pending",
                input: {},
                raw: "",
              },
            })
            toolcalls[value.id] = part as MessageV2.ToolPart
            break

          case "tool-call":
            // 工具调用执行（参数完整）
            const match = toolcalls[value.toolCallId]
            if (match) {
              const part = await Session.updatePart({
                ...match,
                tool: value.toolName,
                state: {
                  status: "running",
                  input: value.input,
                  time: { start: Date.now() },
                },
                metadata: value.providerMetadata,
              })
              toolcalls[value.toolCallId] = part as MessageV2.ToolPart
            }
            break

          case "tool-result":
            // 工具执行结果（状态完成）
            const match = toolcalls[value.toolCallId]
            if (match) {
              await Session.updatePart({
                ...match,
                state: {
                  status: "completed",
                  input: value.input,
                  output: value.output.output,
                  title: value.output.title,
                  time: { start: match.state.time.start, end: Date.now() },
                },
              })
            }
            delete toolcalls[value.toolCallId]
            break

          case "finish":
            // 流式输出结束
            break
        }
      }
    } catch (e: any) {
      // 错误处理和重试
      log.error("process", { error: e, stack: JSON.stringify(e.stack) })
      const error = MessageV2.fromError(e, { providerID: input.model.providerID })
      const retry = SessionRetry.retryable(error)
      if (retry !== undefined) {
        attempt++
        const delay = SessionRetry.delay(attempt, error.name === "APIError" ? error : undefined)
        SessionStatus.set(input.sessionID, {
          type: "retry",
          attempt,
          message: retry,
          next: Date.now() + delay,
        })
        await SessionRetry.sleep(delay, input.abort).catch(() => {})
        continue
      }
      input.assistantMessage.error = error
      Bus.publish(Session.Event.Error, {
        sessionID: input.assistantMessage.sessionID,
        error: input.assistantMessage.error,
      })
    }

    // 返回是否继续循环
    if (needsCompaction) return "compact"
    if (blocked) return "stop"
    if (input.assistantMessage.error) return "stop"
    return "continue"
  }
}
```

## 六、核心设计原则

### 1. 增量处理

- **原则**：来了一个事件处理一个，不等待完整数据
- **优势**：实时反馈，用户体验好

### 2. 状态驱动

- **原则**：通过状态机管理数据从"不完整"到"完整"的转换
- **优势**：可追踪、可恢复、可调试

### 3. 实时保存

- **原则**：每次增量都同步到数据库，不等到完整才保存
- **优势**：数据安全，支持多端同步

### 4. 事件驱动

- **原则**：响应式处理，每个事件触发相应动作
- **优势**：解耦，可扩展

## 七、常见问题

### Q1: 为什么不等待完整数据再处理？

**A**: 流式输出的核心价值在于**实时反馈**。如果等待完整数据，用户会长时间看到白屏或加载中，体验很差。

### Q2: 如果网络中断，已处理的数据会丢失吗？

**A**: 不会。每次增量都通过 `Session.updatePart` 保存到数据库，已处理的部分数据不会丢失。

### Q3: 如何知道工具调用是否完整？

**A**: 通过状态机。`tool-input-start` → `tool-call` → `tool-result`，只有收到 `tool-result` 才算完整。

### Q4: 为什么 `tool-input-delta` 事件不处理？

**A**: 在当前实现中，`tool-input-delta` 事件是空处理。工具参数在 `tool-call` 事件中一次性完整接收，而不是逐步累积。这个事件保留是为了未来的扩展可能。

### Q5: 为什么需要 `toolcalls` 对象？

**A**: 因为流式事件是无序的，可能收到 `tool-call` 但还没收到对应的 `tool-input-start`。`toolcalls` 对象维护了当前待处理的工具调用映射。

### Q6: 增量累积会不会性能问题？

**A**: 不会。增量是字符串拼接，时间复杂度 O(n)，且每次增量都是小片段（通常 1-10 个 token），性能开销可忽略。

## 八、总结

OpenCode 处理 LLM 流式输出的核心机制：

1. **接收位置**：在内层循环中通过 `for await (const value of stream.fullStream)` 接收
2. **处理策略**：增量累积 + 状态机管理
3. **保存机制**：每次增量都实时保存到数据库，UI 实时显示
4. **状态转换**：pending → running → completed（工具调用）
5. **用户体验**：实时看到进度，而不是等待完整响应

这种设计使得 OpenCode 能够提供流畅、可靠的用户体验，同时保证数据的完整性和可恢复性。
